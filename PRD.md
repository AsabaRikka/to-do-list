# 待办清单 PRD（全栈版 / 截图对齐）

## 1. 产品概述

核心定位：聚焦用户在线整理待办清单的核心需求，提供简单、高效的任务管理与列表管理能力，并对齐参考截图的布局、交互与视觉风格。

本版本从“纯前端本地存储”升级为“前后端分离”，支持账号与多列表（待办集），为后续数据同步与扩展打基础。

## 2. 目标与非目标

### 2.1 目标（MVP）
- 支持账号登录（注册可选），左侧展示用户信息。
- 支持任务管理：新增/编辑/完成/删除/星标/备注（备注多行）。
- 支持“智能分类”筛选：今日/全部/已完成/未完成/星标，并显示计数。
- 支持“待办集”管理：创建/重命名/删除待办集，点击切换右侧任务列表，并显示计数。
- 支持搜索：在当前视图结果内按标题/备注过滤。
- UI 布局与视觉严格对齐参考截图（两列固定布局、卡片样式、按钮位置与密度）。

### 2.2 非目标（本期不做）
- 多端实时同步冲突解决、离线编辑队列。
- 权限协作/共享列表的完整链路（可先保留“分享”按钮为占位或复制摘要）。
- 复杂统计面板与报表。

## 3. 页面结构与交互规范（与参考截图 1:1 对应）

### 3.1 整体布局规则
- 页面分为两列固定排布：
  - 左侧列：固定宽度（导航与分类栏），不随内容挤压变形。
  - 右侧列：宽度自适应（核心内容展示区）。
- 删除独立的数据统计模块：任务计数直接嵌入左侧分类/待办集标题后（如“任务 5”“周工作安排 3”）。
- 移动端（<md）：左侧列以抽屉（drawer）形式出现，右侧内容优先展示。

### 3.2 左侧列（固定宽度，从上到下）

#### 3.2.1 用户信息区（截图左上）
- 展示：头像、昵称、邮箱。
- 搜索框：保留，样式与截图一致。

#### 3.2.2 任务分类模块（截图左侧上部，横向标签）
- 模块标题：任务分类
- 分类项（横排展示，可横向滚动）：
  - 今日任务、全部任务、已完成、未完成、星标任务
- 交互：点击分类项 → 右侧任务列表自动切换到对应视图并筛选任务。
- 视觉：标签式设计；选中项高亮，未选中项弱化，与整体风格统一。
- 计数（写死口径）：
  - 默认显示“未完成数”（剩余任务数）
  - “已完成”分类显示完成数
  - “今日/星标”按其筛选集合的未完成数

#### 3.2.3 待办集模块（截图左侧下部列表）
- 模块标题：待办集
- 列表项样式：包含列表图标、名称、任务计数（如“周工作安排 3”），视觉对齐截图。
- 交互：
  - 点击某待办集 → 右侧显示该列表下所有任务
  - 支持创建/重命名/删除
- 删除策略（写死）：
  - 删除待办集时级联删除该列表下所有任务
  - 必须二次确认（避免误删）
- “新建列表”按钮：
  - 放置在模块最底部，样式与截图一致
  - 点击后进入创建流程（行内输入或弹层均可，最终效果一致）

### 3.3 右侧列（自适应宽度）

#### 3.3.1 顶部：列表标题区（截图右上）
- 展示当前选中的分类/待办集名称（例如“周工作安排”）。
- 右上角功能按钮：分享、更多操作（可先实现基础交互或占位）。

#### 3.3.2 主体：任务列表模块（截图右侧主体）
- 任务项样式严格对齐截图：
  - 圆形复选框、任务标题、星标按钮
  - 卡片圆角、间距、阴影/描边、密度与截图一致
- 支持：勾选完成、星标切换、编辑、删除（删除建议二次确认）。
- 备注展示：
  - 备注以小号字体显示在标题下方
  - 点击备注文字进入编辑态，支持多行，编辑框高度随内容自动调整
- “添加任务”入口：
  - 放置在列表最底部，样式与截图一致（“+ 添加任务”条）

## 4. 任务展示与排序规则（固定）
- 排序规则（统一且可预测）：
  1) 未完成在前，已完成在后
  2) 同组内按 deadline：有截止时间优先；越近越靠前；无 deadline 放最后
  3) 再按 createdAt 倒序（新 → 旧）

## 5. 技术栈建议（本地优先，后续可扩展）

### 5.1 前端
- React + TypeScript + Vite
- Tailwind CSS
- 状态管理：Zustand（单一数据源 + selectors）

### 5.2 后端
- Node.js + Express
- 用户认证：JWT（Authorization: Bearer <token>）

### 5.3 数据库
- 本地开发：SQLite（文件库，零依赖，启动快）
- 未来生产：可切换为 MySQL（保持 API 与模型一致，走迁移工具/ORM）

### 5.4 本地存储角色
- 前端仅保存 token/少量 UI 状态（如最后选中的视图、搜索词）。
- 任务/待办集数据以服务端为准；前端可做缓存但不是权威数据源。
